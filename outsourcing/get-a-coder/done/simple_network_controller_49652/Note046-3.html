Definitions, acronyms and abbreviations in DAQ/EF prototype -1

1 Definitions

Back-end system
All the software for configuring, controlling and monitoring of the DAQ prototype -1. It
excludes: the management, processing and transportation of physics data.

Collector
An element of a sub-farm which obtains events from the processing elements of the sub-farm and provides them to the SFO.

Crate control
A high-level DAQ function of the LDAQ which allows an operator to coordinate the activities of IOMs within a DAQ-unit. A more detailed definition is given in [1] .

Crate fragment
The set of ROB fragments, within a ROC, with the same Gid and buffered within the L3IF.

Crate identifier (Cid)
The unique system wide identifier of a ROC.

Concurrent Versions System (CVS)
The version control system provides the possibility to record the history of file modifications and retrieve previous versions [2] .

Configuration databases
The configuration databases store the parameters necessary to describe the DAQ system's
architecture, hardware and software components, running modes and status [3] .

Data collection
The collecting and possible ordering of a set of ROB fragments into a crate fragment.

Data control messages
The data, excluding ROB or crate fragments, exchanged between IOMs. They cause the
receiving IOM to perform a specific action.

Data Flow Manager (DFM)
A element of the event building sub-system which ensures the correct flow of data between
event builder sources and destinations. A more detailed definition is given in [4] .

Dataflow system
All software and hardware required for the management, transportation and monitoring of
physics data.

DAQ-unit
The functionality of the read-out crate or the sub-farm crate concerned with the management and transportation of physics data.

Data Viewing System (DVS)
Allows users to have access to aggregate displays, such as histograms and trace plots, repre
senting the performance of sub-systems of the data acquisition system [5] .

Destination identifier (Did)
The unique system wide identifier of a destination. This is a high-level identifier, independ
ent of technology.

Detector event type (provisional)
An identifier, unique within a sub-detector, which identifies the data within a ROB fragment, e.g. a ROB fragment may contain calibration or monitoring data.

Diagnostic package
This element uses the tests held in the test manager to diagnose problems with the DAQ and
confirm its functionality [6] .

Distributor
An element of a sub-farm which obtains events from the SFI and distributes them to the
processing elements within it's sub-farm.

Event
All ROB fragments with the same Gid. Alternatively, all crate fragments with the same Gid.

Event Builder (EB)
The sub-system which provides the coordinated transfer of ROB or crate fragments between event builder sources and destinations.

Event builder Destination
A element of the event building sub-system which receives data from the event building net
work. A more detailed definition is given in [4] .

Event builder Source
A element of the event building sub-system which puts ROB or crate fragments into the
event building network. A more detailed definition is given in [4] .

Event Handler
The logical object consisting of an event distributor, an event collector, one or more processing elements, an event handler supervisor and an appropriate communication layer.

Event Filter (EF) system
The hardware and software required for the on-line processing of all physics data. A more
detailed definition is given in [7] .

Event Filter Interface (EFIF)
An instance of an IOM. It receives data control messages from the trigger module and col
lects ROB fragments, from the ROBs within its ROC, to form a crate fragment. The crate or
ROB fragments are subsequently input into the Event building sub-system. A more detailed
definition is given in [8] .

Event sampling
A high-level DAQ function of the LDAQ which provides crate or ROB fragments to applications mounted in the back-end. A more detailed definition is given in [1] .

Event filter supervisor
The hardware and software required to globally control the Event Filter. It is also responsible for the configuration, initialisation and overall error handling of the Event Filter.

Event fragment
A generic term for a sub-set of event data. Specific instances of an event fragment are a ROB fragment and a crate fragment.

Event type
A unique ATLAS trigger/DAQ wide identifier which classifies an event, e.g. physics, cali
bration, monitoring.

Event Viewing System (EVS)
Provides the facility for viewing event data in all detectors, together with a description of the event classification [9] .

Generic I/O module
An element, incorporating both software and hardware, which encapsulates the common core functionality of the I/O modules. A more detailed definition is given in [10] .

Global event Identifier (Gid)
The unique, for a run, DAQ wide identifier of an event.

I/O module (IOM)
An element, incorporating both software and hardware, within a DAQ-unit which receives,
buffers and outputs: ROB fragments and or crate fragments and or data control messages.
More specifically it is a ROB or TRG or L3IF or SFI. A more detailed definition is given in
[8] .

Level 1 trigger (L1)
The first processing system, within the trigger/DAQ, which rejects or accepts events based
on an analyses of a sub-set of an event's data.

Level 2 trigger (L2)
A processing system which rejects or accepts events based on an analyses of a sub-set of an
event's data. It analyses those events accepted by the Level 1 trigger.

Level three interface (L3IF)
See Event filter interface.

Local DAQ (LDAQ)
The element which provides the high-level DAQ functions, not related to the main flow of
data, within the Dataflow. It also provides the interface to the back-end. A more detailed def
inition is given in [1] .

Message Reporting System (MRS)
A facility which allows all software components in the ATLAS DAQ system to report error
messages to other components [11] .

Network controller
The technology specific hardware and software which initialises, configures, controls and
monitors the event building network.

Network interface
The technology specific hardware and software which connects modules to the event build
ing network.

Objectivity/DB
This is a fully distributed object orientated database used for the configuration, versions and
backup databases [12] .

Object Kernel Support (OKS)
This is a package which provides a simple active persistent in-memory object manager which is used to implement run-time configuration databases [13] .

Partition
The physical and or logical separation of one or more elements of the experiment into mutu
ally exclusive sub-sets. A more detailed definition is given in [14] .

Partition Identifier (PID)
The unique identifier of a partition. This is a high-level identifier, independent of technology.

Partition manager
Formalises the allocation of hardware and software elements, allowing people to work in
parallel without interference [15] .

Process manager
This element performs basic job control of software components of the DAQ (starting, stop
ping and monitoring basic status) [16] .

Processing Element
A logical object consisting of one or more processing task and the processing node on which they are running.

Processing Node
The hardware on which run one or more processing tasks.

Processing Task
The software running on a processing node in charge of transforming the event. The transformation may be filtering, reformatting, monitoring or calibration. Several processing tasks can be running concurrently on the same processing node.

Read-Out Buffer (ROB)
An instance of an IOM. It receives and buffers ROB fragments from a ROD via a ROL and
provides the data to the level two trigger system and Event Filter. It also receives data control messages from other IOMs within its ROC. A more detailed definition is given in [8] .

Read-Out Crate (ROC)
The combined functionality of the LDAQ, TRG, L3IF and one or more ROBs.

Read-Out Driver (ROD)
The element of the detector specific electronics which, via the ROL, sends ROB fragments to a ROB [17] .

Read-Out Link (ROL)
The physical link between the ROD and ROB.

ROB fragment
The data exchanged between the ROD and ROB.

Run bookkeeper
An element which archives information about the data recorded to permanent storage by the
DAQ system [18] .

Run Control
Controls the data taking activities by coordinating the operating of the DAQ sub-systems,
back-end software components and external systems [19] .

Source identifier (Sid)
The unique system wide identifier of a source. This is a high-level identifier, independent of
technology.

Software Release Tools (SRT)
These tools are used to manage the development of multiple concurrent versions of software packages [20] .

Status display
An element which presents the status of the current data taking run to the user in terms of its
main run parameters, detector configuration, trigger rate and state of run controller [21] .

Software through Pictures (StP)
This is the Software through Pictures for the Object Modelling Technique. It is a graphical,
object-oriented modelling environment that helps people work together to analyse and design critical systems [22] .

Sub-Farm
The basic component of the Event Filter. It comprises the Event Handler and the Sub-Farm DAQ [23] .

Sub-Farm Crate (SFC)
The combined functionality of the LDAQ, the switch to farm interface and Sub-farm output. A more detailed definition is given in [23] .

Sub-Farm Input
See Switch to farm interface.

Sub-Farm Output (SFO)
An instance of an IOM. It receives (and possibly buffers) events from the Event Filter and
sends them to one or more storage systems. A more detailed definition is given in [23] .

Event Handler Supervisor
The object in charge of the control of the event handler. It provides initialization, configuration and run control functionalities. It also provides monitoring information to the Event Filter supervisor.

Switch to farm interface (SFI)
An instance of an IOM. It receives and buffers all the event fragments to form an event and
provides the event to the Event Handler. A more detailed definition is given in [8] .

Test manager
Provides a means of organising individual tests for hardware and software components [24] .

Tools.h++
The Rogue Wave Tools.h++ is an internationalised C++ foundation class library which con
tains basic data types and container classes, includes support for exception and multi-
threaded programs, allows to use Endian streams for exchanging binary data among Win
dows, OS/2 and Unix environments, and provides non-intrusive persistence for objects [25] .

Trigger module (TRG)
An instance of an IOM. It receives data control messages from the system and sends them to
other IOMs within its ROC. A more detailed definition is given in [8] .
